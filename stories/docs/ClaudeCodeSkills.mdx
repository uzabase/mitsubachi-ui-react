# Claude Code の使い方

mitsubachi-ui-react では、[Claude Code](https://claude.ai/claude-code) をコンポーネント開発の主要ツールとして活用しています。<br/>
このページでは、プロジェクトで使用している Claude Code のカスタマイズ機能と、その使い分けについて説明します。

## CLAUDE.md / Skills / カスタムスラッシュコマンド

Claude Code のプロジェクトカスタマイズには主に 3 つの仕組みがあります。

### 1. CLAUDE.md — プロジェクトの共通ルール

`CLAUDE.md` はリポジトリルートに置く設定ファイルで、会話のたびに自動で読み込まれます。プロジェクト全体に適用される基本ルール向きですが、長くしすぎると効果が薄れるため要点を絞って記述します。

**このプロジェクトでの使い方:**

- コーディング規約（CSS は class セレクタのみ、論理プロパティ必須、`:focus-visible` など）
- コミットメッセージのフォーマット
- ディレクトリ構造の概要
- 開発コマンド一覧

### 2. Skills — タスク特化の詳細プロンプト

Skills は `.claude/skills/<name>/SKILL.md` に配置するファイルで、`/スキル名` で明示的に呼び出します（常時読み込みではありません）。CLAUDE.md に書くと長くなりすぎる詳細な手順を、タスクごとに網羅的に記述できます。

**このプロジェクトでの使い方:**

- `/story` — ストーリーファイルの作成手順（擬似状態の網羅方法、コンポーネント分類ごとの構成など）
- `/create-storybook-docs` — MDX ドキュメントの作成手順（セクション構成、テンプレート）
- `/design-system-review` — コードレビューの観点と手順（10 の観点、重大度レベル）

### 3. カスタムスラッシュコマンド — Skills の別名

カスタムスラッシュコマンド（`/story`、`/create-storybook-docs` など）は Skills を呼び出すためのインターフェースです。<br/>
`.claude/skills/` 配下の SKILL.md がそのまま対応するスラッシュコマンドになります。

---

## 実際の使い分け

| 仕組み    | 配置場所                    | 読み込みタイミング       | 向いている内容         |
| --------- | --------------------------- | ------------------------ | ---------------------- |
| CLAUDE.md | リポジトリルート            | 毎回自動                 | 短く簡潔なルール・規約 |
| Skills    | `.claude/skills/*/SKILL.md` | `/コマンド名` で明示的に | 長く詳細なタスク手順   |

**補足:**

- CLAUDE.md には「何を守るか」を書き、Skills には「どうやるか」を書くイメージです
- CLAUDE.md から `ai-guide/` 配下のドキュメントを参照する構成にしており、CLAUDE.md 自体は簡潔に保っています
- Skills 内からも `ai-guide/` を参照することで、ルールの重複を避けています

## 利用可能なカスタムスラッシュコマンド

### `/story` — ストーリーファイル作成

コンポーネントに対して、標準パターンに従った `.stories.tsx` ファイルを自動生成します。

**使いどころ:**

- 新しいコンポーネントを作成した後
- 既存コンポーネントにストーリーを追加したいとき

**生成されるもの:**

- 擬似状態のカバレッジ（Normal / Hover / Active / Focus / Disabled）
- バリアント別ストーリー
- AllStates / AllVariants 一覧ストーリー
- JSDoc による各ストーリーの説明

**コンポーネントの種類に応じた構成:**

| 分類             | 例                    | ストーリー構成                            |
| ---------------- | --------------------- | ----------------------------------------- |
| 単一状態         | IconButton, SearchBox | 擬似状態 + バリアント別 + AllStates       |
| 二値状態         | Checkbox, FilterChip  | 状態値 × 擬似状態のマトリクス + AllStates |
| インタラクティブ | Dialog, Snackbar      | 開閉操作を含むストーリー                  |
| 複合             | Table, Timeline       | 子コンポーネントの組み合わせ              |

**使用例:**

```
/story IconButton のストーリーを作成して
```

---

### `/create-storybook-docs` — MDX ドキュメント作成

コンポーネントの MDX ドキュメントページを自動生成します。API 仕様だけでなく、「いつ使うか」「いつ使わないか」を明確にしたガイドを作ります。

**使いどころ:**

- コンポーネントの実装が完了した後
- ドキュメントが未作成のコンポーネントにドキュメントを追加したいとき

**生成されるもの:**

- コンポーネントの概要説明
- **When to Use** — このコンポーネントを使うべきシーン
- **When NOT to Use** — 代替コンポーネントへの誘導付き
- **Service Context** — Speeda AI Agent での具体的な使い方
- **バリエーション** — Canvas ブロックでのビジュアルプレビュー
- **アクセシビリティ** — 対応状況
- **Props** — ArgTypes による Props ドキュメント
- **関連コンポーネント** — リンク付きの関連コンポーネント一覧

**使用例:**

```
/create-storybook-docs Snackbar のドキュメントを作成して
```

---

### `/design-system-review` — デザインシステムレビュー

コンポーネントのコードを、デザインシステムの規約に基づいて厳密にレビューします。PR 作成前のセルフチェックに活用できます。

**使いどころ:**

- コンポーネントの実装が完了したとき
- プルリクエスト作成前のコードチェック
- 既存コンポーネントの品質改善

**レビュー観点（10 項目）:**

| 観点                | チェック内容                                   |
| ------------------- | ---------------------------------------------- |
| API 設計と DX       | Props の命名、デフォルト値、JSDoc              |
| コンポーネント設計  | 単一責務、再利用性、コンポーザビリティ         |
| 状態伝播            | disabled/error 状態の子コンポーネントへの伝播  |
| アクセシビリティ    | WCAG 2.2、ARIA、フォーカス管理                 |
| CSS 設計            | class セレクタ、論理プロパティ、:focus-visible |
| デザイントークン    | トークンの正確な使用、生値の排除               |
| TypeScript 型安全性 | 正確な型定義、strict mode 準拠                 |
| アイコン            | 既存アイコンの再利用、命名規則                 |
| パフォーマンス      | 不要な再レンダリング、バンドルサイズ           |
| テスト容易性        | ユニットテスト、ストーリーパターン             |

**重大度レベル:**

- 🔴 Critical — 必ず修正が必要
- 🟠 Important — 強く推奨
- 🟡 Moderate — 推奨
- 🔵 Minor — 改善提案

**使用例:**

```
/design-system-review 今の変更をレビューして
```

## スキルの活用フロー

新規コンポーネント開発の典型的なフローは以下の通りです：

1. コンポーネントを実装する
2. `/story` でストーリーファイルを生成
3. `/create-storybook-docs` で MDX ドキュメントを生成
4. `/design-system-review` で品質チェック
5. レビュー指摘を修正し、PR を作成
