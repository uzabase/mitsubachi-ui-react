# デザインシステムレビュースキル

このスキルは、デザインシステムとReactに精通したフロントエンドエキスパートの視点で、mitsubachi-ui-reactプロジェクトのコードを厳しくレビューする。過去のレビューで繰り返し発見された問題パターンを熟知しており、同じ過ちを見逃さない。

## Use when:

- ユーザーが `/review` コマンドを実行した時
- コンポーネントの実装が完了し、レビューを求められた時
- プルリクエスト作成前のコードチェック時
- ユーザーが明示的にコードレビューを依頼した時

## レビュー手順

レビューは以下の順序で実施する。手順を飛ばさないこと。

1. **ai-guideを参照する**: `ai-guide/` 配下のドキュメントを読み、プロジェクトの実装規約を把握する。特に以下は必読：
   - `ai-guide/components.md` — コンポーネント設計パターン
   - `ai-guide/styling.md` — CSS設計方針の詳細
   - `ai-guide/coding-rules.md` — 命名規則・TypeScriptルール
   - `ai-guide/design-tokens.md` — 利用可能なトークン一覧
2. **変更されたファイルを特定する**: `git diff` や `git status` で変更対象を把握する
3. **各ファイルの内容を読み込む**: 変更ファイルだけでなく、関連するCSS・テスト・Storybookも読む
4. **下記の観点で厳しくレビューする**: 各観点のチェック項目を1つずつ確認する
5. **具体的な改善提案をコード例付きで示す**
6. **良い点も積極的に指摘する**

## レビュー観点

### 1. API設計と開発者体験（DX）

コンポーネントを**使う側**の体験を最優先で考える。

- **直感的なAPI**: propsの命名は分かりやすいか？必要最小限か？
- **適切なデフォルト値**: 設定されているか？JSDocの `@default` で明示されているか？
- **学習コスト**: 新しい開発者がすぐに使い始められるか？必須propsが多すぎないか？
- **JSDocコメント**: Props interfaceに日本語のJSDocが書かれているか？冗長な説明（prop名の繰り返し）を避けているか？

**よくある問題:**

- `showXxx` ブール値 + `xxx` 値のセット指定を要求するAPI。値の有無で表示を判定できるなら `showXxx` は不要。利用者に2つのpropsを常にセットで渡させるのは冗長
  ```tsx
  // NG: showCount + maxCount のセット指定
  <TextArea showCount maxCount={100} />
  // OK: maxCount の有無で判定
  <TextArea maxCount={100} />
  ```
- 合成コンポーネントでHTML属性（`required`, `disabled`等）を独自propsとして再宣言している。`extends HTMLAttributes` で既に持っている属性を再宣言すると、destructure時に `...rest` から消えてネイティブ要素に到達しなくなる
  ```tsx
  // NG: requiredを再宣言
  interface Props extends Omit<HTMLAttributes<...>, ...> {
    required?: boolean; // ← HTMLの属性を二重定義
  }
  // OK: rest経由でネイティブ要素に渡す
  const { label, ...rest } = props;
  <textarea {...rest} /> // requiredが自然に渡る
  ```

### 2. コンポーネント設計

- **単一責任の原則**: コンポーネントの責務は明確か？
- **再利用性**: 異なるコンテキストでも使えるか？
- **composability**: 他のコンポーネントと組み合わせやすいか？
- **制約の適切さ**: 過度に制約的でも、過度に自由でもないか？

**禁止事項:**

- **className propの外部公開**: デザインシステムのコンポーネントは閉じたスタイリングを持つ。利用側からのスタイル上書きは一貫性を破壊するため、`className` propが存在する場合は**必ず指摘し、削除を求める**
- **ベースコンポーネントにバリアント固有スタイルを入れる**: ベースのCSSクラスには全バリアント共通のスタイルのみ定義する。一部バリアントにしか必要ないスタイル（gap, padding等）をベースに入れると、不要なバリアントに意図しない余白やレイアウトが生まれる

### 3. 状態の伝播

合成コンポーネント（複数の子コンポーネントを組み合わせたもの）では、`disabled` や `error` などの状態が**すべての構成要素に視覚的に波及する**ことを確認する。部分的な状態適用は一貫性を損なう。

**チェックポイント:**

- disabled時にラベル・補足テキスト・カウント表示など**すべての子要素**がdisabledスタイルになっているか
- disabled時にアイコンの色も `--text-disabled` や `--icon-disabled` トークンに切り替わっているか。子コンポーネントのアイコン色はCSS変数オーバーライドで制御する
- 子要素が明示的に `color` を持つと親のdisabled色がCSS継承で上書きされる。`.parentDisabled .child { color: ... }` のセレクタが必要

### 4. アクセシビリティ

セマンティックHTMLによって一定レベルのアクセシビリティを確保する（WCAG 2.2準拠）。

- **セマンティックHTML**: 適切なHTML要素を使用しているか？（`<button>`, `<label>`, `<textarea>` 等）
- **キーボード操作**: キーボードだけで操作できるか？
- **ARIA属性**: 必要なARIA属性が適切に設定されているか？
- **フォーカス管理**: フォーカス状態が適切に管理されているか？

**よくある問題:**

- フォームのエラーメッセージに `role="alert"` はあるが、`aria-describedby` で入力要素と紐づけていない。`role="alert"` はエラー出現時の読み上げ用、`aria-describedby` はフォーカス時にエラー内容を伝えるために**両方必要**。`useId()` でIDを生成し、エラー有無で `aria-describedby` を動的に設定する
  ```tsx
  const errorId = useId();
  <textarea aria-describedby={errorMessage ? errorId : undefined} />
  {errorMessage && <span id={errorId} role="alert">{errorMessage}</span>}
  ```

### 5. CSS設計

`ai-guide/styling.md` に完全準拠しているかを厳密にチェックする。

**必須ルール:**

- **class名セレクタのみ**: IDセレクタ・タグ名セレクタは使わない
- **論理プロパティの徹底**: `margin`, `padding` のショートハンドも物理プロパティ扱い。全辺同値でも `padding-block` + `padding-inline` に分けて書く
  ```css
  /* NG */ padding: 8px;
  /* OK */ padding-block: 8px; padding-inline: 8px;
  ```
- **`:focus-visible` のみ**: `:focus` は使わない。2リングシャドウ（白内環 + #191919外環）を適用する
- **擬似クラスで状態管理**: `:hover:not(:disabled)`, `:active:not(:disabled)`, `:disabled`
- **PCファースト + 720pxブレイクポイント1箇所のみ**
- **デザイントークン**: 色・スペーシング・フォントサイズ等は必ず `var(--token, fallback)` 形式で参照する。生の値（`8px`, `#333` 等）を直書きしない

**よくある問題:**

- `letter-spacing` をpx固定値で指定している。フォントサイズの比率で指定すべきなので `em` 単位を使う（例: `0.02em`）
- アイコンサイズにviewport別の指定がない。desktopとphoneでアイコンサイズが異なることが多いため、メディアクエリでviewport別に `inline-size` / `block-size` を指定する
- `all: unset` 後に `box-sizing: border-box` と `display` の再宣言が漏れている

### 6. デザイントークンの正確性

トークンの値を**推測しない**。

- Figmaで明示的に指定された値を忠実に反映する。「よくある値」「一般的なデフォルト」に引っ張られない（例: アイコン=24pxと思い込んでFigmaの20px指定を無視する）
- トークン名からフォールバック値を推測しない。`--spacing-small` が `4px` か `8px` かはトークン定義次第
- `ai-guide/design-tokens.md` でトークンの実際の値を確認する

### 7. TypeScript型定義

- **型の正確性**: 型定義が実装と一致しているか？
- **型の厳密性**: `any` や不適切な `as` を避けているか？
- **ポリモーフィックコンポーネントのref型**: `as` propで要素タイプが変わる場合、refのキャストには `Ref<HTMLElement>`（スーパータイプ）ではなく `Ref<HTMLButtonElement & HTMLSpanElement>`（intersection型）を使う。`Ref<T>` のTは共変ではないため、スーパータイプへのキャストは型エラーになる
- **Props interfaceの命名**: `XxxProps` 形式で、コンポーネントと同じファイルに定義する（別ファイルに分離しない）

### 8. アイコン

- 新規アイコンを作る前に `src/icons/` の既存アイコンを確認し、用途が近いものがあればそれを使う
- アイコンコンポーネント名にサイズ情報を含めない（`CheckSmallIcon` → `CheckIcon`）。サイズはCSS側の責務
- SVGパスは推測せず、デザイナーから提供されたベクターデータを使う。`fill="black"` 等のハードコードは `currentColor` に置き換える

### 9. パフォーマンス

- **不要な再レンダリング**: memo化やuseCallbackが適切に使われているか？
- **バンドルサイズ**: 不要な依存関係を含んでいないか？
- **レンダリングコスト**: 重い処理がレンダリング内で実行されていないか？

### 10. テスト容易性

- ユニットテストが書きやすい構造か？
- 必要に応じてdata-testid等が設定されているか？
- Storybookストーリーは標準パターン（擬似状態ごとのstory）に従っているか？
- AllStates系ストーリーは、バリアントを列・状態を行にしたグリッドで構成されているか？

## 致命度の定義

各指摘には以下の致命度を必ず付与する：

| 致命度 | ラベル | 意味 | マージ可否 |
|--------|--------|------|-----------|
| 🔴 Critical | 致命的 | バグ・セキュリティ・アクセシビリティ違反・機能破壊。これがあるとユーザーに直接被害が出る | マージ不可。必ず修正 |
| 🟠 Important | 重要 | デザインシステム規約違反・顕著なDX問題・型安全性の欠如。放置すると負債化する | 原則修正。正当な理由があればスキップ可 |
| 🟡 Moderate | 中程度 | コード品質・保守性・パフォーマンス上の懸念。今は動くが将来問題になりうる | 推奨。判断は開発者に委ねる |
| 🔵 Minor | 軽微 | 命名の好み・より良い書き方の提案・微細なスタイル改善 | 任意。好みの範囲 |

## 出力フォーマット

各ファイルごとに以下の形式でレビュー結果を出力する：

```markdown
## ファイル名:行番号

### 🔴 Critical / 🟠 Important / 🟡 Moderate / 🔵 Minor：[指摘タイトル]

**観点**: [該当する観点名]
**重要度**: [高 / 中 / 低]

[具体的な指摘内容]

\`\`\`tsx
// 現在のコード
[問題のあるコード]

// 改善案
[改善されたコード]
\`\`\`

<!-- トレードオフがある場合のみ以下を記載 -->
> **修正方針のトレードオフ**
>
> | | メリット | デメリット |
> |---|---------|-----------|
> | 改善案A | ... | ... |
> | 改善案B | ... | ... |
>
> **推奨**: [どちらを推奨するか、その理由]

---
```

### 出力ルール

- 致命度は**すべての指摘に必ず付与**する（良い点 ✅ には不要）
- 良い点は従来通り `### ✅ 良い点：[タイトル]` で記載する
- トレードオフの表は**判断が分かれる場合のみ**記載する（自明な修正には不要）
- トレードオフがある場合は複数の選択肢を示し、それぞれのメリット・デメリットを具体的に書く
- 最後に推奨案とその理由を明記する（判断を開発者に丸投げしない）

## レビューサマリー

レビューの最後に以下のサマリーを出力する：

```markdown
## 📊 レビューサマリー

| 致命度 | 件数 |
|--------|------|
| 🔴 Critical | N件 |
| 🟠 Important | N件 |
| 🟡 Moderate | N件 |
| 🔵 Minor | N件 |
| ✅ 良い点 | N件 |

**総合判定**: 🔴 マージ不可 / 🟠 要修正 / 🟡 軽微な改善推奨 / ✅ 問題なし
```

## レビュー完了チェックリスト

すべてのチェック項目を確認してからレビュー結果を提出する：

- [ ] `ai-guide/` のドキュメントを参照したか
- [ ] `className` prop を外部に公開していないか
- [ ] `showXxx` + 値のセット指定など冗長なAPIがないか
- [ ] HTML属性を独自propsで再宣言していないか
- [ ] disabled/error状態が全構成要素に伝播しているか（アイコン色含む）
- [ ] `aria-describedby` でエラーメッセージと入力要素が紐づいているか
- [ ] CSSは論理プロパティのみか（`padding` ショートハンドも不可）
- [ ] `:focus-visible` を使用しているか（`:focus` は不可）
- [ ] class名セレクタのみ使用しているか
- [ ] デザイントークンを `var(--token, fallback)` 形式で使っているか（生の値は不可）
- [ ] `letter-spacing` は `em` 単位か
- [ ] アイコンサイズにviewport別の指定があるか
- [ ] PCファーストで720pxのブレイクポイントか
- [ ] ポリモーフィックrefにintersection型を使っているか
- [ ] ベースコンポーネントに全バリアント共通でないスタイルが混入していないか
- [ ] 不要な再レンダリングを避けているか
- [ ] `any` や不適切な `as` 使用を避けているか
- [ ] Storybookストーリーは擬似状態ごとに用意されているか
